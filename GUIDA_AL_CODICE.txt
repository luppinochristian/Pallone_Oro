1. backend/config/db.php  (~637 righe)
─────────────────────────────────────────────────────────────────────────────
RUOLO: Cuore del backend. Incluso da tutti gli altri file PHP tramite
       require_once. Si occupa del database e delle funzioni condivise.

COSA FA:
  • Apre (o crea) il file SQLite in backend/data/db.sqlite.
  • Alla prima esecuzione, crea automaticamente tutte le tabelle
    del gioco: accounts, players, teams, leghe, nazioni, notizie,
    lega_risultati, champions_bracket, player_skills, e molte altre.
  • Popola il database con i dati iniziali: oltre 200 squadre suddivise
    in più di 20 campionati appartenenti a 15+ nazioni.
  • Definisce le funzioni usate ovunque nel backend:

  FUNZIONI CHIAVE:
    getDB()               → restituisce la connessione PDO al database
                            (singleton: una sola connessione per richiesta)
    getAuthPlayerId()     → legge il token dall'header X-Auth-Token,
                            lo verifica nella tabella account_tokens e
                            restituisce l'ID del giocatore autenticato
    createAccountToken()  → genera un token sicuro con random_bytes(32)
                            e lo salva nella tabella account_tokens
    getLang()             → legge l'header X-Lang dalla richiesta HTTP
                            e restituisce 'en' o 'it' (default 'it')
    t($it, $en)           → helper bilingue: restituisce il testo nella
                            lingua corrente della richiesta
    sendMail($to,$sub,$body) → invia email via PHP mail()
    _getCachedBody()      → decodifica il body JSON della richiesta una
                            sola volta e lo mette in cache (evita letture
                            multiple di php://input)

PATTERN USATO:
  Il database viene creato con CREATE TABLE IF NOT EXISTS, quindi
  db.php è sicuro da chiamare più volte: non sovrascrive nulla
  se il database esiste già.

COME STUDIARLO:
  Inizia da getDB() per capire la connessione PDO + SQLite.
  Poi guarda lo schema delle tabelle (le CREATE TABLE) per capire
  come è strutturato il modello dati del gioco.

2. backend/api/auth.php  (~585 righe)
─────────────────────────────────────────────────────────────────────────────
RUOLO: Gestisce tutto ciò che riguarda account utente e sessioni.

COSA FA:
  • Riceve richieste POST con il campo "action" nel body JSON.
  • In base all'action, chiama la funzione corrispondente:

  FLUSSO REGISTRAZIONE:
    1. doRegister()     → valida email/username/password
                          controlla unicità nel DB
                          genera codice OTP a 6 cifre
                          salva in email_verifications (scade in 15 min)
                          invia email con il codice
                          risponde con { pending_verification: true }

    2. doVerifyCode()   → verifica che il codice OTP corrisponda
                          e non sia scaduto
                          crea l'account nella tabella accounts
                          genera un token di sessione
                          risponde con { token, email, username }

    3. doResendCode()   → genera un nuovo OTP e lo invia
                          (il vecchio diventa invalido)

  FLUSSO LOGIN:
    doLogin()           → verifica email/password con password_verify()
                          genera un nuovo token
                          risponde con { token, careers: [...] }

  GESTIONE CARRIERE:
    getCareers()        → lista le carriere dell'account autenticato
    createCareer()      → crea un nuovo giocatore (player) con team iniziale
    deleteCareer()      → elimina un giocatore e tutti i suoi dati
    renameCareer()      → aggiorna il nome del giocatore

  EMAIL TRANSAZIONALI:
    sendVerificationEmail() → email con codice OTP, bilingue IT/EN
    sendResetEmail()        → email con link reset password, bilingue

PATTERN IMPORTANTE:
  Ogni funzione fa output diretto con echo json_encode(...) e poi return.
  Non usa classi o eccezioni per il controllo del flusso: è PHP procedurale
  semplice e diretto.

3. backend/api/game.php  (~2311 righe) — IL FILE PIÙ IMPORTANTE
─────────────────────────────────────────────────────────────────────────────
RUOLO: Motore completo del gioco. Contiene tutta la logica di simulazione.

COSA FA (funzione principale: playMonth):
  Quando il giocatore preme "Gioca Mese", questa funzione:
    1. Verifica l'autenticazione e carica i dati del giocatore
    2. Applica gli effetti delle azioni scelte (allenamento, riposo...)
    3. Controlla infortuni (probabilità basata sulle azioni)
    4. Calcola variazione morale, energia, popolarità
    5. Simula le partite di campionato del mese
    6. Simula (se qualificato) le fasi della Champions Cup
    7. Aggiorna l'overall del giocatore
    8. Calcola stipendio + bonus
    9. Genera 3-8 notizie dinamiche sul mese appena passato
    10. (a giugno) Calcola Pallone d'Oro, promozione/retrocessione,
        resetta la stagione e crea il prossimo calendario
    11. Salva tutto nel database e risponde con i dati aggiornati

  FUNZIONI DI SIMULAZIONE:
    simulaGiornataLega()        → simula tutte le partite del mese per
                                   tutti i team della lega. Chiama
                                   simulaPartita() per le partite NPC.
    simulaPartita($t1,$t2)      → simula una singola partita tra due team
                                   usando i loro overall per calcolare
                                   il risultato (V/P/S) e il punteggio
    simulaChampions()           → gestisce la progressione nella Champions
                                   Cup: gironi → ottavi → quarti → semi → finale
    simulaGiornataGruppo()      → partite nella fase a gironi
    simulaFaseChampions()       → partite nelle fasi a eliminazione diretta
    concludiChampions()         → assegna il trofeo al vincitore

  GENERAZIONE NOTIZIE:
    generaNotizieDinamiche()    → la funzione più lunga del file (~500 righe).
                                   Analizza i dati della partita appena giocata
                                   (gol, assist, voto, esito, avversario) e
                                   sceglie tra 400+ template di notizie.
                                   Ogni notizia è bilingue (IT/EN) tramite _tn().
    getAdvancedMatchNews()      → notizie dettagliate sulla singola partita
    getExtraNewsPool()          → notizie "extra" su forma, morale, mercato

  FUNZIONI HELPER:
    getPlayerData($id)          → carica tutti i dati del giocatore con JOIN
                                   su teams, leghe, nazioni
    getPlayerSkills($db,$id)    → lista delle abilità sbloccate
    simulateMatch($p,$skills)   → simula la performance individuale del
                                   giocatore (gol, assist, voto) per una partita
    getBonusStruttura($livello) → moltiplicatore attributi per livello struttura
    calcPalloneDoro()           → algoritmo multicriteria per il Pallone d'Oro
    checkPromozioneRetrocessione() → promuove/retrocede a fine anno
    applySkillBoost()           → sblocca un'abilità dall'Albero delle Abilità

  FUNZIONI BILINGUE:
    _tn(string $it, string $en)         → sceglie IT o EN in base a getLang()
    _tna(array $it, array $en)          → come _tn() ma per array

COME STUDIARLO:
  Inizia da playMonth() e segui il flusso. È la funzione "orchestratrice"
  che chiama tutte le altre in sequenza. Poi approfondisci simulaGiornataLega()
  per capire la matematica del gioco.

9. router.php  (~44 righe)
─────────────────────────────────────────────────────────────────────────────
RUOLO: Router per il server PHP built-in usato in sviluppo locale.

COSA FA:
  • Se la richiesta è per un file statico esistente (CSS, JS, immagini),
    lo serve direttamente (return false fa gestire il file al server).
  • Se la richiesta è per un file PHP nel backend, lo include ed esegue.
  • Per tutto il resto, serve frontend/index.html (routing SPA).

  Usato SOLO con: php -S localhost:8080 router.php
  In produzione (Apache/Nginx) non serve: il web server gestisce il routing.

10. frontend/index.html  (~240 righe di markup, più script tags)
─────────────────────────────────────────────────────────────────────────────
RUOLO: Unico file HTML del progetto. È una Single Page Application (SPA).

STRUTTURA:
  L'HTML definisce tutti i "pannelli" del gioco come <div class="page">.
  Solo quello con classe "active" è visibile. JavaScript cambia quale
  pannello è attivo senza mai ricaricare la pagina.

  PANNELLI PRINCIPALI:
    #auth-page           → login / registrazione / verifica OTP
    #career-select-box   → selezione carriera dopo login
    #game-page           → interfaccia principale di gioco
    #skill-page          → albero delle abilità
    #agent-page          → gestione agente
    #notizie-page        → storico notizie
    #standings-page      → classifica campionato
    #champions-page      → tabellone Champions Cup
    #objectives-page     → obiettivi e missioni
    #transfer-page       → mercato trasferimenti
    #epilogo-page        → schermata fine carriera

  ORDINE CARICAMENTO SCRIPT (importante!):
    I moduli vengono caricati in ordine. app.js deve essere ULTIMO
    perché usa le funzioni definite negli altri file.
    gamedata → locale → commentary → achievements → season_planner →
    tutorial → ui_components → analytics → timeline → particles →
    crowd → charts → encyclopedia → animations → playercard → app.js

  PATTERN data-i18n:
    Gli elementi HTML con data-i18n="chiave" vengono tradotti
    automaticamente da _applyTranslations() in app.js quando
    si cambia lingua. Esempio:
      <button data-i18n="auth_login">Accedi</button>
    Dopo cambio a inglese diventa "Login" automaticamente.

16. frontend/js/app.js  (~5054 righe) — IL FILE PRINCIPALE
─────────────────────────────────────────────────────────────────────────────
RUOLO: Cervello del frontend. Coordina tutta l'applicazione client-side.

ORGANIZZAZIONE INTERNA (sezioni in ordine):

  [1] STRINGHE UI BILINGUI (righe ~1-600)
      Un grande oggetto STRINGS con due sottooggetti: 'it' e 'en'.
      Contiene centinaia di chiavi per ogni testo dell'interfaccia.
      La funzione _t(key) recupera la stringa nella lingua corrente.

  [2] SISTEMA DI TRADUZIONE NOTIZIE (righe ~600-690)

PATTERN IIFE (JavaScript):
  Tutti i moduli JS usano il pattern IIFE (Immediately Invoked Function Expression):
    const GS_NomeModulo = (() => {
        // variabili private
        function privata() { ... }
        // API pubblica
        return { funzione1, funzione2 };
    })();
  Questo crea un scope privato per le variabili interne ed espone
  solo le funzioni pubbliche nell'oggetto restituito.

API WRAPPER — la funzione api() in app.js:
  Tutte le chiamate al backend passano per api(file, data, method).
  Aggiunge automaticamente: token di autenticazione, lingua, gestione errori.
  Restituisce il JSON parsato o null in caso di errore.

BILINGUE IT/EN — come funziona ovunque:
  PHP:        t('testo italiano', 'english text')
              _tn($it, $en) per stringhe in notizie dinamiche
  JavaScript: _t('chiave_stringa') — recupera da STRINGS[currentLang]
  HTML:       data-i18n="chiave" — aggiornato da _applyTranslations()

SQLITE SENZA ORM:
  Il progetto usa PDO direttamente con query SQL parametrizzate (?).
  Non usa alcun ORM. Ogni file PHP fa le proprie query.
  Le prepared statement proteggono dalle SQL injection.

SINGLE PAGE APPLICATION senza router:
  Non c'è React Router o Vue Router. La navigazione è gestita manualmente:
  ogni pagina è un div, showPage('nome') fa display:none su tutti e
  display:block su quello richiesto. L'URL non cambia (niente history API).

CANVAS NATIVO senza librerie:
  Particelle, folla, grafici e radar sono tutti disegnati con

  GIORNO 1 — Capire la struttura:
    • Leggi questo documento dall'inizio
    • Apri index.html e guarda la struttura dei div
    • Leggi db.php per capire il modello dati
    • Apri il gioco nel browser e usa i DevTools per vedere le richieste API

  GIORNO 2 — Il ciclo di gioco:
    • Leggi playMonth() in game.php (la funzione più importante)
    • Segui la chiamata da app.js: cerca "play_month" in app.js
    • Guarda come i dati ritornati aggiornano renderGame()

  GIORNO 3 — Autenticazione:
    • Leggi auth.php dall'inizio alla fine
    • In app.js cerca doRegister(), showVerifyPanel(), doVerifyCode()
    • Capire il flusso OTP completo

  GIORNO 4 — Frontend avanzato:
    • Studia il pattern IIFE guardando ui_components.js (il più breve)
    • Poi guarda achievements.js per capire localStorage
    • Infine particles.js per capire il canvas animation loop

  GIORNO 5 — Internazionalizzazione:
    • Leggi locale.js e gamedata.js
    • In app.js cerca _translateNewsItems() per capire la traduzione AI
    • Prova a cambiare lingua nel browser e osserva il comportamento
